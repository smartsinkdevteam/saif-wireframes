비활성화하면 새 주문 배정 후보에서 제외돼요. 기존에 배정된 작업과 과거 이력은 그대로 유지돼요.

⚠ 진행 중인 배정 작업 2건이 있어요. 비활성화해도 이 작업들은 계속 진행돼요.
케이스 4 — 저장 완료 수정 후 [A-3.3 기사 상세]로 복귀 + 완료 토스트
✓ 기사 정보가 수정되었습니다. B

구성 (A-3.2 기사 등록과 동일 폼 · 수정 전용 차이만)

  1. 1 진입: 기사 상세(A-3.3)의 [수정] → 이 화면(IA에 A-3.4 신설). 저장/취소 시 상세로 복귀. 등록(A-3.2)과 동일한 2섹션(계정 정보 / 추가 정보) 구조.
  2. 2 휴대전화번호: 등록에선 본인 변경 불가였지만, 관리자 수정 화면에서는 변경 가능. 로그인 아이디라 변경 시 기사 안내 필요 경고. 중복 시 유효성 에러(케이스 2).
  3. 3 비밀번호: 수정 화면은 기존 비밀번호를 표시하지 않음. 등록의 "초기 비밀번호 입력" 대신 [비밀번호 초기화](임시 비밀번호 재생성)로 대체.
  4. 4 담당 업무(실측/시공): 기존 선택값 채워짐. 권한 분기 직결(실측=입금 확인 가능, 시공=일정·결제 미노출).
  5. 5 취급 가능 품목: 다품목 반영 — 냉장고장·붙박이장·신발장·싱크대 체크박스 그룹(기존 선택값 채워짐). ※ 등록(A-3.2)은 현재 "냉장고장" 1개만 노출 → 등록도 동일 4종 그룹으로 맞추는 것을 권장(아래 동기화 항목).
  6. 6 상태(활성/비활성): 등록엔 없는 수정 전용 필드(신규는 항상 활성). 비활성은 신규 배정 후보 제외, 기존 배정·이력 유지.
  7. 7 사업자등록증: 기존 파일이 있으면 [보기]·[교체]·[삭제], 없으면 등록과 동일한 업로드 영역.
  8. 8 [저장]: 유효 시 반영 → [A-3.3]로 복귀, 완료 토스트(케이스 4). 상태를 비활성으로 바꿨으면 케이스 3 확인 모달 거침.
  9. 9 휴대전화번호 중복: "이미 등록된 휴대전화번호입니다." (등록과 동일 문구).
  10. A 비활성화 확인 모달: 진행 중 배정이 있으면 건수 경고. 삭제가 아니라 soft 비활성(주문·이력 참조 보존).
  11. B 완료 토스트: "기사 정보가 수정되었습니다." 후 상세 복귀.

⚠️ 명세/IA 동기화 · 확정 필요

  1. A-3.4를 기능 명세서에도 추가(현재 IA에만 신설). 등록(A-3.2)과 동일 폼이되, ① 휴대전화번호 변경 허용+경고 ② 초기 비밀번호 입력 → [비밀번호 초기화] ③ 상태(활성/비활성) 필드 추가 ④ 사업자등록증 [보기·교체·삭제]가 수정 전용 차이.
  2. 취급 품목 4종 그룹: 다품목 반영으로 등록·수정 모두 냉장고장·붙박이장·신발장·싱크대 체크박스 그룹이 되어야 일관됨. 등록(A-3.2) 와이어프레임도 동일하게 갱신 권장.
  3. 담당 업무 단일/복수(실측·시공 겸임), 전화번호 변경 시 기존 로그인 세션 처리, 비활성 기사의 진행 중 작업 재배정 정책 — 확정 필요.

⚠️ 문서에 없는 임의 표현

  1. 등록 화면의 CSS·레이아웃·섹션 구분을 그대로 사용. 상태 필드·[비밀번호 초기화]·파일 [보기/교체/삭제] 버튼 배치, 비활성화 모달 문구·샘플 값은 임의 표현 — 확정 필요.
  2. 샘플 데이터(이름·번호·주소·차량)는 전부 가상의 예시.
