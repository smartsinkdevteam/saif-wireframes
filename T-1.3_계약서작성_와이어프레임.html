취소 작성 완료 I

명세 반영 사항 (기능 명세서 T-1.3 기준)

  1. 1 진입: [T-1.1.1 주문 상세]의 [계약서 작성] 액션(모든 품목 측정완료 후 활성). 위저드 외부 별개 화면.
  2. 2 견적 행(다품목 반영): 주문의 시공범위 품목이 한 행씩 자동 구성 — 여러 종류면 종류별 행(실측 시점 확정 품목 기준 = 주문 등록 시공범위 + 현장 추가·취소 반영). 품목명 읽기 전용, 행 직접 추가·삭제 불가. 기사는 품목별 금액만 입력(0 이상 정수, 천 단위 구분자, 품목당 최대 1억원). 모델은 인스턴스당 1행(order_item 기준).
  3. 3 합계: 전 품목 금액 자동 합산(읽기 전용).
  4. 4 작성일: 계약서 작성(오늘) 날짜 자동 기록(읽기 전용).
  5. 5 기사 서명: 휴대폰 화면에 손으로 직접 그리는 서명 캔버스(별도 외부 라이브러리 없이 화면 내 캔버스). 필수.
  6. 6 [다시 그리기]: 서명 캔버스 초기화 후 재입력 — 기사·고객 서명 캔버스 모두 동일 제공.
  7. 7 고객 서명: 동일한 서명 캔버스, 선택 입력. 받는 즉시 입력.
  8. 8 고객 서명이 비어 있으면 [작성 완료] 시 계약 "보류"(미완료), 있으면 "확정" — 명세 그대로(안내 문구는 임의).
  9. 9 [취소]: 작성 중 상태 유지하고 [T-1.1.1 주문 상세]로 복귀.
  10. A [작성 완료] 활성 조건: 모든 품목 금액 입력 + 기사 서명 완료(고객 서명 없어도 활성). 확정 시 착수금 수령 처리로 결제 상태 "착수금 대기"→"잔금 대기" 전이.
  11. B 확인 모달 → 확인 시 계약서 PDF/이미지 자동 생성 → [T-1.1.1] 복귀. 확정본은 기사 휴대폰에 jpg/pdf 저장·다운로드 가능. 계약서에는 품목별 금액·기사/고객 서명·착수금 입금 내역이 포함.
  12. C PDF 생성 대기 중 로딩 표시(생성 실패 시 에러 토스트 — 케이스 5).
  13. D 고객 서명 없이 완료 → 보류 저장. [실측 완료]는 계약 "확정"이어야 활성(보류면 비활성) — 명세 비고 그대로.
  14. E 보류 재진입: 입력값(금액·기사 서명) 유지된 채 열림 → 고객 서명 받고 재[작성 완료] 시 확정.
  15. F 확정 후 현장에서 품목 추가·취소 시 확정 자동 무효화 → [계약서 작성] 재활성·재작성(6/5 결정, T-1.1.1에서 처리).
  16. G 금액 미입력·음수: "금액을 0 이상으로 입력해주세요." — 명세 문구 그대로(필드 하단). 여러 품목 중 빈 행만 에러 표시.
  17. H 토스트: "모든 품목의 금액을 입력해주세요." / "기사 서명을 입력해주세요." / "계약서 생성에 실패했습니다. 잠시 후 다시 시도해주세요." — 명세 문구 그대로.
  18. I 활성 조건 미충족 시 [작성 완료] 비활성(시각적 dim).

⚠️ 문서에 없는 임의 표현 (와이어프레임 작성자 판단)

  1. 1차도 한 주문에 여러 종류 품목을 받을 수 있어(같은 종류 1개씩) 케이스 1·5의 다품목 행(냉장고장·붙박이장·신발장)은 실제 1차 동작에 해당. 같은 종류 여러 개·별칭은 추후(데이터 모델만 선반영).
  2. 상단 고객·주소 요약 칩 — 명세에 없으나 계약 대상 확인용으로 배치(제외 가능).
  3. 보류/확정 안내 배너(케이스 1·4)와 모달 문구 일체 — 동작은 명세 기준, 문구는 임의(확정 필요).
  4. 서명 푸터의 "실측 기사 · OOO / 고객 · OOO" 표기, 금액 placeholder("금액 미입력")는 임의.
  5. 비활성 [작성 완료] + 토스트 병행 표시 방식(버튼 비활성 vs 탭 시 토스트)은 명세 미정의 — 둘 다 표현, 확정 필요.
  6. [다시 그리기]는 기사·고객 서명 모두 제공 — 명세 표에는 기사 서명에만 명시되어 있어 동기화 필요(06-12 결정).
